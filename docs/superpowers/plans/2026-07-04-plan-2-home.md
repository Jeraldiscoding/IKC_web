# IKC Website — Plan 2: Home Page + Route Stubs

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the temporary one-screen Home placeholder with a full, long-scrolling, SEO-optimized landing page assembled from focused section components, and add lightweight SEO stub pages for the five other routes so site navigation no longer 404s.

**Architecture:** The Home page (`app/page.tsx`) becomes a thin Server Component that composes a sequence of self-contained section components from `src/components/home/`. All display copy lives in typed data modules under `src/content/` (single source of truth, reused by later Services/FAQ plans). Decorative visuals are lightweight inline-SVG components in `src/components/illustrations/`. Every page carries its own `metadata` (per-page SEO) and the Home page adds `WebSite` + `FAQPage` JSON-LD.

**Tech Stack:** Next.js 16 App Router (Server Components, static generation), Tailwind v4 tokens from Plan 1, lucide-react icons, existing UI primitives (`Button`, `Card`, `Badge`, `Section`) and SEO helpers (`pageMetadata`, `JsonLd`) from Plan 1. Tests: Vitest + React Testing Library.

## Global Constraints

- All app code under `frontend/`; work on branch `plan-2-home`.
- Reuse Plan 1 primitives — do NOT recreate `Button`/`Card`/`Badge`/`Section`, the SEO helpers, or `site-config`. Import them.
- Business data (WhatsApp `8023 1551` → `wa.me/6580231551`, address/email placeholders, nav) comes ONLY from `@/lib/site-config`. Never hardcode these.
- Every page: exactly one `<h1>`; semantic `<h2>`/`<h3>` hierarchy; keyword-rich but natural headings.
- Palette tokens only: `cream`/`cream-dark`, `terracotta`/`terracotta-dark`/`terracotta-light`, `mustard`/`mustard-dark`, `sage`/`sage-dark`, `ink`/`ink-muted`. Fonts: `font-heading` (Baloo 2), `font-body` (Inter).
- No fabricated testimonials, credentials, prices, dates, or fake parent quotes. Placeholder business facts stay clearly generic.
- Section components are Server Components (no `"use client"`) unless they need interactivity — none in this plan do.
- Primary SEO keywords to weave naturally into headings/copy: "enrichment classes for children with special needs in Singapore", "home-based special needs enrichment centre", "small-group SPED classes", "DISE-certified special needs educator (NIE)", "functional skills training", "school holiday programmes", "hands-on sensory learning".
- TDD adapted (same as Plan 1): real tests for content/behavior contracts (a section renders its heading + key copy, data modules have the right shape, pages expose one h1); `next build` as the smoke-gate for pure layout.

export function ActivityGlyph({ className = "" }: { className?: string }) {
  return (
    <svg
      viewBox="0 0 120 120"
      className={className}
      aria-hidden="true"
      focusable="false"
      fill="none"
      stroke="currentColor"
      strokeWidth="4"
      strokeLinecap="round"
      strokeLinejoin="round"

      <circle cx="34" cy="34" r="12" />
      <path d="M34 12v-6M34 62v-6M12 34H6M62 34h-6M18 18l-4-4M50 50l4 4M50 18l4-4M18 50l-4 4" />
      {/* building blocks */}
      <rect x="60" y="66" width="22" height="22" rx="3" />
      <rect x="84" y="66" width="22" height="22" rx="3" />
      <rect x="72" y="44" width="22" height="22" rx="3" />
    </svg>
  );
}
```

*Why abstract SVG, not stock photos or cartoon characters:* it renders crisp at any size, carries no licensing risk, never looks like a broken image, and reads as intentional brand decoration. Real class photos/videos slot in later without redesign.

import { siteConfig } from "@/lib/site-config";
import { faqs } from "@/content/faqs";

export function faqPageSchema() {
  return {
    "@context": "https://schema.org",
    "@type": "FAQPage",
    mainEntity: faqs.map((f) => ({
      "@type": "Question",
      name: f.question,
      acceptedAnswer: { "@type": "Answer", text: f.answer },
    })),
  };
}

export function webSiteSchema() {
  return {
    "@context": "https://schema.org",
    "@type": "WebSite",
    name: siteConfig.name,
    url: siteConfig.url,
  };
}
```

*Why FAQPage schema on the Home page:* the Home FAQ preview uses real Q&A content, so marking it up lets Google show those questions as rich results — one of the concrete gaps we found in every competitor site.
